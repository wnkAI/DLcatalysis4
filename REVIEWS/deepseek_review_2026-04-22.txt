## 1. Tensor Shape Consistency at Int3DCrossAttn Boundaries

Looking at `int3d_cross_attn.py`:

```python
# Expected shapes:
# pocket_feats: [B, K, D] where K=32 (pocket residues)
# atom_feats: [B, A, D] where A=variable (substrate atoms)
# pocket_xyz: [B, K, 3]
# atom_xyz: [B, A, 3]
# xyz_valid: [B] boolean mask

# In forward():
def forward(self, pocket_feats, atom_feats, pocket_xyz, atom_xyz, xyz_valid):
    B, K, D = pocket_feats.shape
    _, A, _ = atom_feats.shape
    
    # Compute pairwise distances
    # pocket_xyz: [B, K, 1, 3]
    # atom_xyz: [B, 1, A, 3]
    # dist: [B, K, A]
    
    # RBF embedding: [B, K, A, R] where R=rbf_bins=32
    
    # Linear projections for Q, K, V
    # Q: [B, K, H, D//H] from pocket_feats
    # K: [B, A, H, D//H] from atom_feats  
    # V: [B, A, H, D//H] from atom_feats
    
    # Attention scores: [B, H, K, A]
    # Add RBF bias: [B, H, K, A] (broadcast from [B, K, A, R] → [B, H, K, A])
    
    # Output: [B, K, D] (updated pocket features)
```

**Shape Issues Found:**
1. **In `v4_pocket.py` line ~180**: The RBF bias addition expects `[B, H, K, A]` but RBF is `[B, K, A, R]`. Need to permute/reshape: `rbf_bias = rbf.sum(dim=-1).unsqueeze(1)` to get `[B, 1, K, A]` then broadcast to `[B, H, K, A]`.
2. **In `v4_ultimate.py` line ~250**: Same issue persists. The code shows `attn = attn + rbf_bias` but `rbf_bias` shape is `[B, K, A, R]` while `attn` is `[B, H, K, A]`.

**Fix needed**: `rbf_bias = self.rbf_embed(dist).sum(dim=-1).unsqueeze(1)` to match attention shape.

## 2. 'at least one sample with xyz_valid' Gating Logic

In `v4_pocket.py` line ~175 and `v4_ultimate.py` line ~245:

```python
if xyz_valid.any():
    # Compute 3D cross-attention
    pocket_feats = self.int3d_cross_attn(
        pocket_feats, atom_feats, pocket_xyz, atom_xyz, xyz_valid
    )
```

**Problem**: This gates the ENTIRE batch based on `xyz_valid.any()`. If batch has 10 samples and only 1 has valid coordinates, ALL 10 samples get the 3D cross-attention computation. The invalid samples will have zero/NaN coordinates that propagate through distance calculations.

**Consequence**: Zero-coord geometry DOES leak between samples. Distance calculations for invalid samples will be nonsense (likely NaN or very large values).

**Fix needed**: Need to mask invalid samples during distance computation or handle them separately.

## 3. PCC Loss Div-by-Zero Safety

In training loss calculation (not shown in diff but in training code):

```python
# Pearson Correlation Coefficient loss
pcc_loss = 1.0 - torch.corrcoef(...)[0, 1]
```

**Issue**: `torch.corrcoef` can return NaN if variance is zero (constant predictions). No clamping or epsilon protection.

**Fix needed**: Add epsilon to denominator: `cov/(std_pred*std_target + eps)`.

## 4. Ranking Loss Gradient Flow

Ranking loss uses pairwise comparisons within batch. Potential issues:
- Gradient flow through `torch.max(margin - diff, 0)` - ReLU-like, gradients stop when `diff > margin`
- Sampling of pairs might create disconnected computation graph if no valid pairs meet `min_diff` threshold
- No gradient flow through the pair selection logic itself (which is correct)

**Verdict**: Mathematically correct but could benefit from gradient clipping for stability.

## 5. DRFP Zeros Fallback

In `v4_ultimate.py` line ~150:

```python
if self.use_rxn_drfp:
    drfp_idx = batch.drfp_idx  # [B]
    drfp_feat = self.drfp_embedding(drfp_idx)  # [B, D]
else:
    drfp_feat = torch.zeros(B, self.drfp_dim, device=device)
```

**Problem**: Zero-initialized DRFP features get added to the prediction through the output header. Since weights are randomly initialized, zeros don't guarantee neutral contribution - they create random noise that varies by batch.

**Degradation**: Yes, this silently degrades the model. Zero features still pass through linear layers with non-zero weights/bias.

**Fix needed**: Either:
1. Gate the entire DRFP branch with a learned gate that can shut it off
2. Use a separate pathway that can be zeroed out without affecting other features

## 6. Annotation Bag-of-Words Mean Pool with All-Zero Padding

In annotation processing:

```python
# interpro_fam: [B, max_fam=16] with 0 for padding
# interpro_emb: [B, max_fam, embed_dim]
# Mean pool: interpro_emb.mean(dim=1)
```

**Issue**: Padding zeros contribute to the mean, diluting the signal. If an enzyme has 3 annotations in a padded tensor of length 16, the mean is over 16 entries (3 real + 13 zeros).

**Fix needed**: Use mask during mean: `(interpro_emb * mask.unsqueeze(-1)).sum(dim=1) / mask.sum(dim=1, keepdim=True)`.
